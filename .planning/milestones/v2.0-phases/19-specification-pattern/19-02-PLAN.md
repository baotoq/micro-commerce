    1. Create `ActiveOrdersSpec.cs`:
       - File-scoped namespace: `MicroCommerce.ApiService.Features.Ordering.Application.Specifications`
       - Sealed class extending `Specification<Order>` from `Ardalis.Specification`
       - Uses `MicroCommerce.ApiService.Features.Ordering.Domain.Entities` for Order
       - Uses `MicroCommerce.ApiService.Features.Ordering.Domain.ValueObjects` for OrderStatus
       - Defines a static readonly array of terminal statuses: `private static readonly OrderStatus[] TerminalStatuses = [OrderStatus.Failed, OrderStatus.Cancelled];`
       - Constructor takes an optional `string? statusFilter = null` parameter
       - Base behavior: `Query.Where(o => !TerminalStatuses.Contains(o.Status));` — excludes terminal statuses (Failed/Cancelled) as the core "active orders" filter
       - Optional narrowing: if `statusFilter` is not null and `OrderStatus.TryFromName(statusFilter, ignoreCase: true, out OrderStatus? status)` succeeds, add `Query.Where(o => o.Status == status);` — narrows to a specific active status
       - This matches REQUIREMENTS.md QUERY-03 which specifies ActiveOrdersSpec, and aligns with ROADMAP success criterion 3
       - Pattern from 19-RESEARCH.md Pattern 5 (ordering spec with terminal status exclusion)

       - Sealed class extending `Specification<Order>`
       - Constructor takes `Guid buyerId` parameter
       - Body: `Query.Where(o => o.BuyerId == buyerId);`
       - Single-responsibility: filters by buyer only; active-order filtering is composed via And() with ActiveOrdersSpec

    Both specs follow the pattern established in Plan 01: filtering only (no ordering, no pagination, no projection). Ordering and pagination stay in handlers.

    1. Refactor `GetAllOrdersQueryHandler.Handle()`:
       - Add usings: `using Ardalis.Specification;`, `using Ardalis.Specification.EntityFrameworkCore;`, `using MicroCommerce.ApiService.Features.Ordering.Application.Specifications;`
       - Replace the inline status filtering with ActiveOrdersSpec:
         ```
         ActiveOrdersSpec spec = new(request.Status);

         IQueryable<Order> query = context.Orders
             .AsNoTracking()
             .WithSpecification(spec);
         ```
       - ActiveOrdersSpec handles both the base "exclude terminal statuses" filter and the optional status narrowing via its statusFilter constructor parameter
       - Keep `OrderByDescending(o => o.CreatedAt)` in handler (sorting is handler concern)
       - Keep Skip/Take pagination in handler
       - Keep the `.Select(o => new OrderSummaryDto(...))` projection in handler unchanged
       - Keep totalCount query structure, applying WithSpecification for consistency

    2. Refactor `GetOrdersByBuyerQueryHandler.Handle()`:
       - Add same usings as above
       - Replace inline buyer + status filtering with spec composition:
         ```
         Specification<Order> spec = new OrdersByBuyerSpec(request.BuyerId);
         spec = spec.And(new ActiveOrdersSpec(request.Status));

         IQueryable<Order> query = context.Orders
             .AsNoTracking()
             .WithSpecification(spec);
         ```
       - This demonstrates And() composition in ordering domain (buyer + active orders with optional status)
       - ActiveOrdersSpec always excludes terminal statuses; the optional statusFilter narrows further
       - Keep OrderByDescending, Skip/Take, Select projection in handler unchanged

    3. Use explicit types throughout per C# conventions: `IQueryable<Order> query`, `Specification<Order> spec`, `int totalCount`, `List<OrderSummaryDto> items`.

    IMPORTANT: Maintain identical API contract — same OrderListDto/OrderSummaryDto shape, same pagination behavior.
