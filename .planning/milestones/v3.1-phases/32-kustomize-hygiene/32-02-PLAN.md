Add explicit imagePullPolicy to all application deployments and standardize Kubernetes labels across all workloads and services.

Purpose: Standard `app.kubernetes.io/*` labels enable consistent querying, monitoring, and tooling integration. Explicit imagePullPolicy prevents ambiguity about when images are pulled (especially important with local kind images using non-latest tags).

Output: All deployments/statefulsets have standard labels and imagePullPolicy; all services use matching selectors.
</objective>

  <action>
For each deployment/statefulset, apply these changes:

**1. Standard labels** — Replace the simple `app: X` labels with standard Kubernetes labels. Add them to THREE places in each manifest: `metadata.labels`, `spec.selector.matchLabels`, and `spec.template.metadata.labels`. All three MUST match for selectors to work.

Label mapping per workload:

| Workload | name | component | part-of |
|----------|------|-----------|---------|
| apiservice | apiservice | api | micro-commerce |
| gateway | gateway | gateway | micro-commerce |
| web | web | frontend | micro-commerce |
| keycloak | keycloak | auth | micro-commerce |
| otel-collector | otel-collector | observability | micro-commerce |
| aspire-dashboard | aspire-dashboard | observability | micro-commerce |
| rabbitmq | rabbitmq | messaging | micro-commerce |
| postgres | postgres | database | micro-commerce |

The label block for each workload should look like:
```yaml
labels:
  app.kubernetes.io/name: {name}
  app.kubernetes.io/component: {component}
  app.kubernetes.io/part-of: micro-commerce
```

Keep the existing `app: X` label alongside the new ones (for backward compatibility with any scripts or ArgoCD selectors that reference it). So the final labels block is:
```yaml
labels:
  app: {name}
  app.kubernetes.io/name: {name}
  app.kubernetes.io/component: {component}
  app.kubernetes.io/part-of: micro-commerce
```

**2. imagePullPolicy** — Add `imagePullPolicy` to each container spec:

| Workload | Image | imagePullPolicy | Reason |
|----------|-------|-----------------|--------|
| apiservice | apiservice:dev | IfNotPresent | Local kind image |
| gateway | gateway:dev | IfNotPresent | Local kind image |
| web | web:dev | IfNotPresent | Local kind image |
| keycloak | quay.io/keycloak/keycloak:26.0 | IfNotPresent | Pinned version |
| otel-collector | otel/opentelemetry-collector:0.146.0 | IfNotPresent | Pinned version |
| aspire-dashboard | mcr.microsoft.com/dotnet/aspire-dashboard:9.3 | IfNotPresent | Pinned version |
| rabbitmq | rabbitmq:4-management | IfNotPresent | Tagged version |
| postgres | postgres:17 | IfNotPresent | Pinned version |

Place `imagePullPolicy` directly after the `image:` field in the container spec.

IMPORTANT: Do NOT change selectors to use ONLY `app.kubernetes.io/*` labels — keep `app: X` in selectors for safety. The `spec.selector.matchLabels` should include ALL labels that are in `spec.template.metadata.labels`.

For each service YAML, add standard `app.kubernetes.io/*` labels to `metadata.labels` and keep the existing `app: X` in `spec.selector` (services select pods by labels, and since pods keep `app: X`, selectors still work).

For each service, add to `metadata.labels`:
```yaml
labels:
  app: {name}
  app.kubernetes.io/name: {name}
  app.kubernetes.io/component: {component}
  app.kubernetes.io/part-of: micro-commerce
```

Use the same component mapping as Task 1. If a service file has NO `metadata.labels` block, add one. If it has multiple Service resources in the same file (rabbitmq, keycloak, postgres all have a second NodePort service), add labels to both.

The `spec.selector` on services should keep just `app: {name}` — this is sufficient to match pods and adding more labels to selectors unnecessarily restricts matching.
