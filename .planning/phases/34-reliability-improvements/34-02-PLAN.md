**Client-side API URL fix (REL-01):**

The problem: `/api/config` returns `getApiBaseUrl()` which resolves to `http://gateway:8080` (cluster-internal). The browser cannot reach this. The server-side URL (`services__gateway__*`) is correct for SSR/server components, but client-side needs a browser-reachable URL.

1. In `src/MicroCommerce.Web/src/lib/config.ts`, add a new exported function:
```typescript
export function getClientApiBaseUrl(): string {
  // NEXT_PUBLIC_API_URL is the browser-reachable gateway URL
  // In K8s: set to http://localhost:38800 (NodePort)
  // In Aspire: the gateway is proxied, so same origin works
  return process.env.NEXT_PUBLIC_API_URL || "";
}
```

2. In `src/MicroCommerce.Web/src/app/api/config/route.ts`, change to:
```typescript
import { getClientApiBaseUrl } from "@/lib/config";
import { NextResponse } from "next/server";

export async function GET() {
  return NextResponse.json({
    apiBaseUrl: getClientApiBaseUrl(),
  });
}
```
When `NEXT_PUBLIC_API_URL` is empty string, the client-side `fetch` calls will use relative URLs (same-origin), which works in Aspire where gateway proxies everything. In K8s, the env var is set to `http://localhost:38800`.

3. In `infra/k8s/base/web/deployment.yaml`, add a new env var to the container:
```yaml
- name: NEXT_PUBLIC_API_URL
  value: "http://localhost:38800"
```
Add it after the existing `services__gateway__https__0` env var. The `services__gateway__https__0` continues to be used for server-side data fetching (SSR). `NEXT_PUBLIC_API_URL` is used by `/api/config` for client-side browser calls.

**RabbitMQ StatefulSet conversion (REL-02):**

4. Create `infra/k8s/base/rabbitmq/statefulset.yaml` — copy the pod template from the existing deployment.yaml but wrap it in a StatefulSet with a volumeClaimTemplate:

7. Update `infra/k8s/base/rabbitmq/service.yaml`: the existing ClusterIP service (first document) needs `clusterIP: None` to become a headless service for StatefulSet. Keep the NodePort service (second document) unchanged.

Add `startupProbe` to each workload. Startup probes tell K8s to wait for slow-starting containers before activating liveness/readiness probes. Use the same endpoint as the readiness probe but with more generous timing (failureThreshold * periodSeconds = max startup time).

For each deployment/statefulset, add a `startupProbe` block in the container spec, BEFORE the existing `livenessProbe`:

1. **gateway** (`infra/k8s/base/gateway/deployment.yaml`):
```yaml
startupProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 12
```
(allows up to 65s startup)

2. **web** (`infra/k8s/base/web/deployment.yaml`):
```yaml
startupProbe:
  httpGet:
    path: /
    port: 3000
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 12
```
(allows up to 65s startup)

3. **rabbitmq** (`infra/k8s/base/rabbitmq/statefulset.yaml`):
```yaml
startupProbe:
  exec:
    command: ["rabbitmq-diagnostics", "ping"]
  initialDelaySeconds: 10
  periodSeconds: 5
  failureThreshold: 24
```
(allows up to 130s startup — RabbitMQ with Erlang/mnesia is slow)

4. **otel-collector** (`infra/k8s/base/otel-collector/deployment.yaml`):
