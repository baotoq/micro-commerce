  <action>
For each of the 8 workloads (apiservice, gateway, web, keycloak, postgres, rabbitmq, otel-collector, aspire-dashboard):

1. **Create a ServiceAccount YAML** (`serviceaccount.yaml`) in the workload's base directory:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: {workload-name}
  labels:
    app: {workload-name}
    app.kubernetes.io/name: {workload-name}
    app.kubernetes.io/component: {component}
    app.kubernetes.io/part-of: micro-commerce
automountServiceAccountToken: false
```

Use the existing `app.kubernetes.io/component` value from each deployment (api, gateway, frontend, auth, database, messaging, observability).

2. **Update each deployment/statefulset** to add:
   - `spec.template.spec.serviceAccountName: {workload-name}` — reference the dedicated ServiceAccount
   - Container-level `securityContext` block on the container:

```yaml
securityContext:
  runAsNonRoot: true
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  capabilities:
    drop:
      - ALL
```

**Important exceptions for `readOnlyRootFilesystem`:**
- **postgres**: Set `readOnlyRootFilesystem: false` — PostgreSQL needs to write to its data directory and /tmp. It already has a PVC mount but also writes to paths outside the mount.
- **keycloak**: Set `readOnlyRootFilesystem: false` — Keycloak writes theme caches, tmp files, and data to multiple directories.
- **rabbitmq**: Set `readOnlyRootFilesystem: false` — RabbitMQ writes to /var/lib/rabbitmq for mnesia database and other runtime state.

For these three, keep `runAsNonRoot: true`, `allowPrivilegeEscalation: false`, and `capabilities.drop: [ALL]`.

**Also for `runAsNonRoot`:**
- **postgres**: The official postgres image runs as root by default. Add `runAsNonRoot: false` for postgres (it handles privilege dropping internally via `gosu`). Set `runAsUser: 999` (the postgres user) instead — or simply set `runAsNonRoot: false` since the official image manages this internally. Use `runAsNonRoot: false` for postgres to avoid startup failures.

Place the `securityContext` at the container level (under `spec.template.spec.containers[0].securityContext`), NOT at the pod level.

3. **Update each kustomization.yaml** to add `serviceaccount.yaml` to the `resources` list. Keep existing resources in order, add serviceaccount.yaml after the existing entries (deployment.yaml, service.yaml, etc.).

All 8 workloads to process:
