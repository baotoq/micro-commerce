3. Create `src/MicroCommerce.ApiService/Common/Behaviors/ResultValidationBehavior.cs`:
   - Namespace: `MicroCommerce.ApiService.Common.Behaviors`
   - `public sealed class ResultValidationBehavior<TRequest, TResponse> : IPipelineBehavior<TRequest, TResponse> where TRequest : notnull where TResponse : IResultBase`
   - Constructor takes `IEnumerable<IValidator<TRequest>> validators`
   - Handle method:
     - If no validators, call `next()`
     - Run all validators via `Task.WhenAll` (same pattern as existing `ValidationBehavior`)

     - If no failures, call `next()`

   - IMPORTANT: The `where TResponse : IResultBase` constraint from FluentResults is what makes this only activate for Result-returning handlers. The `(TResponse)(object)Result.Fail(errors)` cast works for non-generic `Result`. Document in a code comment that `Result<T>` support requires reflection and is deferred to ADOPT-05.
   - Use explicit types, file-scoped namespace, `sealed class`, follow ValidationBehavior.cs style closely.

2. Create `docs/decisions/adr-006-result-exception-boundary.md`:
   - Title: "ADR-006: Result vs Exception Boundary"
   - Status: Accepted
   - Date: 2026-02-24
   - Context: Explain that the codebase uses throw-based error handling for all failures, and Phase 17 introduces FluentResults for explicit error handling of expected business rule violations.
   - Decision:
     - Business rule violations that callers can reasonably handle → Return `Result.Fail` (HTTP 422 via ToHttpResult)
     - Not-found errors (resource expected to exist) → Throw `NotFoundException` (HTTP 404 via GlobalExceptionHandler)
     - Concurrency conflicts → Throw `ConflictException` (HTTP 409 via GlobalExceptionHandler)
     - Input validation failures (malformed request) → Throw `ValidationException` via ValidationBehavior (HTTP 400)
     - Unexpected exceptions (bugs, infrastructure) → Let propagate to GlobalExceptionHandler (HTTP 500)
   - Consequences: Gradual migration — existing handlers unchanged, new Result-returning handlers opt in. Pilot scope uses non-generic `Result` only; `Result<T>` with ResultValidationBehavior deferred to ADOPT-05.
   - Limitations: `ResultValidationBehavior` cast `(TResponse)(object)Result.Fail(errors)` only works for non-generic `Result`. Pilot handlers should not rely on `Result<T>` in the validation path.
  </action>
