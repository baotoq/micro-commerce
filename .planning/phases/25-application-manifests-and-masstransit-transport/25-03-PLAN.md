**New steps to add (after waiting for infra pods, before access info):**

**Step 9: Build application container images**
```bash
# --- Step 9: Build application container images ---
info "Building application container images..."

info "  Building ApiService..."
dotnet publish "$PROJECT_ROOT/src/MicroCommerce.ApiService/MicroCommerce.ApiService.csproj" \
  /t:PublishContainer \
  -p:ContainerRegistry="" \
  -p:ContainerRepository=apiservice \
  -p:ContainerImageTag=dev \
  --nologo -v quiet

info "  Building Gateway..."
dotnet publish "$PROJECT_ROOT/src/MicroCommerce.Gateway/MicroCommerce.Gateway.csproj" \
  /t:PublishContainer \
  -p:ContainerRegistry="" \
  -p:ContainerRepository=gateway \
  -p:ContainerImageTag=dev \
  --nologo -v quiet

info "  Building Web..."
docker build \
  -f "$PROJECT_ROOT/src/MicroCommerce.Web/Dockerfile" \
  -t web:dev \
  "$PROJECT_ROOT/src/MicroCommerce.Web" \
  --quiet
```

Key decisions:
- `-p:ContainerRegistry=""` overrides the ghcr.io default from Directory.Build.props -- we want local images, not pushed to registry
- `-p:ContainerRepository=apiservice` overrides the baotoq/micro-commerce/apiservice default -- simple name for kind
- `-p:ContainerImageTag=dev` matches the Kustomize overlay images section
- `--nologo -v quiet` reduces build noise
- Web uses `docker build` with its existing Dockerfile

**Step 10: Load images into kind**
```bash
# --- Step 10: Load images into kind ---
info "Loading images into kind cluster..."
kind load docker-image apiservice:dev --name "$CLUSTER_NAME"
kind load docker-image gateway:dev --name "$CLUSTER_NAME"
kind load docker-image web:dev --name "$CLUSTER_NAME"
```

**Step 11: Apply application manifests via dev overlay**
This step is handled by the MODIFIED Step 7 which now uses `kubectl apply -k overlays/dev/` instead of `kubectl apply -k base/`. Since the overlay includes base, all infrastructure AND application manifests are applied together.

However, infrastructure pods need to be ready BEFORE application pods can start (ApiService needs PostgreSQL, RabbitMQ, Keycloak). The existing step 8 waits for infra pods. But images must be loaded BEFORE kubectl apply for apps to start.

**Revised flow:**
1. Steps 1-6: Cluster, SealedSecrets, namespace, seal secrets (unchanged)
2. Step 7: Apply ONLY infrastructure via `kubectl apply -k base/` (keep original -- infra manifests don't reference app images)
3. Step 8: Wait for infra pods (unchanged)
4. Step 9: Build app images (NEW)
5. Step 10: Load images into kind (NEW)
6. Step 11: Apply full dev overlay (NEW) -- `kubectl apply -k "$SCRIPT_DIR/overlays/dev/"`. This is idempotent for infra (already applied in step 7) and adds app deployments.
7. Step 12: Wait for app pods (NEW)
8. Step 13: Print access info (UPDATED)

**Step 11: Apply application manifests**
```bash
# --- Step 11: Apply application manifests ---
info "Applying application manifests..."
kubectl apply -k "$SCRIPT_DIR/overlays/dev/"
```

**Step 12: Wait for app pods**
```bash
# --- Step 12: Wait for application pods ---
info "Waiting for ApiService (may take up to 3 minutes for first-boot migrations)..."
kubectl wait --for=condition=ready pod -l app=apiservice -n "$NAMESPACE" --timeout=180s

info "Waiting for Gateway..."
kubectl wait --for=condition=ready pod -l app=gateway -n "$NAMESPACE" --timeout=120s

info "Waiting for Web..."
kubectl wait --for=condition=ready pod -l app=web -n "$NAMESPACE" --timeout=120s
```
ApiService gets 180s timeout (7 schema migrations on first boot).
