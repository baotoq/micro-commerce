## User Constraints (from CONTEXT.md)

### Locked Decisions

**Migration scope:**
- Migrate **IDs only** — all 13 `StronglyTypedId<Guid>` types (ProductId, OrderId, CartId, etc.) plus EventId in BuildingBlocks.Common
- **Do NOT** touch domain value objects (Money, Quantity, ProductName, CategoryName, DisplayName) — they stay as readonly record structs
- Migrate **all IDs at once** in a single pass — they're all identical Guid-based types, low risk
- **Delete** `StronglyTypedId<T>` base class from BuildingBlocks.Common after migration
- **Delete** `StronglyTypedIdConvention.cs` — rely entirely on Vogen's built-in EF Core ValueConverters
- **Convert EventId** in BuildingBlocks.Common to Vogen (adds Vogen dependency to BuildingBlocks.Common)

**Vogen configuration style:**
- **Per-type attributes** on each ID — each type explicitly declares `[ValueObject<Guid>]` with full config (self-documenting over DRY)
- Generate **EF Core + System.Text.Json** conversions only (no Swagger TypeConverter, no Dapper, no LINQ to DB)
- IDs generated as **record struct** — matches existing readonly record struct convention, value semantics, stack-allocated
- Keep ID definitions in their **existing per-feature Domain/ValueObjects/ folders** — consistent with vertical slice structure

**Validation behavior:**
- **Validate and reject Guid.Empty** — Vogen's `Validate()` method rejects default/empty Guid on creation
- **Throw exception** on validation failure (Vogen's default `ValueObjectValidationException`) — fail-fast, DDD style
- Add **`New()` factory method** using **Guid v7** (`Guid.CreateVersion7()`) on each ID type — e.g., `ProductId.New()` instead of `ProductId.From(Guid.NewGuid())`

### Claude's Discretion

- Exact Vogen attribute syntax and any needed workarounds for EF Core integration
- How to handle the BaseDbContext convention registration change (removing StronglyTypedIdConvention)
- EF Core migration generation approach (whether new migrations are needed or schema is unchanged)

### Deferred Ideas (OUT OF SCOPE)

- Adopting Vogen for domain value objects (Money, Quantity, etc.) — potential future phase if Vogen proves beneficial for IDs
- Adding Swagger/OpenAPI TypeConverter generation — revisit if API schema issues arise

## Summary

Phase 16.1 replaces 14 hand-rolled `StronglyTypedId<Guid>` record types (13 in ApiService + EventId in BuildingBlocks.Common) with Vogen-generated `record struct` equivalents. Vogen 8.0.4 is the current stable release; it is a pure source generator with no runtime dependencies, making installation lightweight. The migration is a pure code transformation — database schema is unchanged because all IDs remain stored as `uuid` columns.

The key mechanical changes are: (1) add `Vogen` NuGet to both `BuildingBlocks.Common.csproj` and `MicroCommerce.ApiService.csproj`, (2) rewrite each ID from a `sealed record` inheriting `StronglyTypedId<Guid>` to a `partial record struct` decorated with `[ValueObject<Guid>]`, (3) delete `StronglyTypedId<T>` base class and `StronglyTypedIdConvention.cs`, (4) update `BaseDbContext.ConfigureConventions` to use Vogen's EF Core auto-registration, and (5) update all 21 call sites using `new XxxId(guid)` constructor syntax to `XxxId.From(guid)`.

**Primary recommendation:** Use `[ValueObject<Guid>(conversions: Conversions.EfCoreValueConverter | Conversions.SystemTextJson)]` per-type with a `Validate()` method rejecting `Guid.Empty` and a `New()` factory using `Guid.CreateVersion7()`. Replace `StronglyTypedIdConvention` with Vogen's `configurationBuilder.RegisterAllInVogenEfCoreConverters()` in `BaseDbContext`. No EF Core migrations are needed because schema (uuid columns) does not change.

|---------|---------|---------|--------------|

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Vogen | Meziantou.Framework.StronglyTypedId | Both are source generators; Vogen has richer validation, assembly-level defaults. Meziantou is lighter. User chose Vogen. |
| Per-type `[ValueObject<Guid>]` | Assembly-level `[VogenDefaults]` | Assembly-level is DRY but hides intent per type. User chose per-type (self-documenting). |


### Pattern 1: ID Type Definition (Vogen record struct)

**What:** Each ID is a `partial record struct` with Vogen attribute, Validate method, and New() factory.

**The exact Vogen attribute syntax (HIGH confidence — verified from Context7 and official docs):**

```csharp
// Source: https://github.com/SteveDunn/Vogen/blob/main/README.md
[ValueObject<Guid>(conversions: Conversions.EfCoreValueConverter | Conversions.SystemTextJson)]
public partial record struct ProductId
{
    public static Validation Validate(Guid value) =>
        value != Guid.Empty
            ? Validation.Ok
            : Validation.Invalid("ProductId cannot be empty.");

    public static ProductId New() => From(Guid.CreateVersion7());
}
```

Key points:
- `partial` is **required** — Vogen generates the other half of the type
- No `readonly` modifier — `record struct` is already value-type; Vogen handles immutability
- No explicit `Value` property — Vogen generates it
- No constructor — Vogen generates `From(Guid)` as the factory
- `New()` is a custom factory method — explicitly defined by us, calls Vogen's generated `From()`
- `Validate()` is a static method Vogen hooks into during `From()` call

**Contrast with current pattern:**

```csharp
// BEFORE (hand-rolled)
[DebuggerStepThrough]
public sealed record ProductId(Guid Value) : StronglyTypedId<Guid>(Value)
{
    public static ProductId New() => new(Guid.NewGuid());
    public static ProductId From(Guid value) => new(value);
}

// AFTER (Vogen-generated)
[ValueObject<Guid>(conversions: Conversions.EfCoreValueConverter | Conversions.SystemTextJson)]
public partial record struct ProductId
{
    public static Validation Validate(Guid value) =>
        value != Guid.Empty
            ? Validation.Ok
            : Validation.Invalid("ProductId cannot be empty.");

    public static ProductId New() => From(Guid.CreateVersion7());
}
```

### Pattern 2: EF Core Convention Replacement

**What:** Replace the hand-rolled `StronglyTypedIdConvention` (IModelFinalizingConvention using reflection) with Vogen's auto-registration.

Vogen generates a nested class `EfCoreValueConverter` inside each ID type. There are two ways to register them:

**Option A: `RegisterAllInVogenEfCoreConverters()` (recommended)**

Requires a marker class decorated with `[EfCoreConverter<T>]` for each ID. This is the "Onion Architecture" approach — the marker class lives in the Infrastructure layer.

```csharp
// In a shared infrastructure file, e.g., Common/Persistence/VogenEfCoreConverters.cs
[EfCoreConverter<ProductId>]
[EfCoreConverter<CategoryId>]
[EfCoreConverter<OrderId>]
// ... all 14 IDs
internal sealed partial class VogenEfCoreConverters;

// In BaseDbContext:
protected override void ConfigureConventions(ModelConfigurationBuilder configurationBuilder)
{
    base.ConfigureConventions(configurationBuilder);
    configurationBuilder.RegisterAllInVogenEfCoreConverters();
    configurationBuilder.Conventions.Add(_ => new AuditableConvention());
    configurationBuilder.Conventions.Add(_ => new ConcurrencyTokenConvention());
    configurationBuilder.Conventions.Add(_ => new SoftDeletableConvention());
}
```

**Option B: Per-property `HasVogenConversion()` in entity configurations**

```csharp
builder.Property(p => p.Id).HasVogenConversion();
```

This is verbose for 14+ IDs across all entities. Option A is preferred.

**Option C: IModelFinalizingConvention with Vogen detection**

A custom convention that detects Vogen-generated types and calls `HasConversion` with the nested `EfCoreValueConverter`. This is more complex but avoids the marker class pattern.

**Recommendation: Option B (per-property `HasVogenConversion()`) is the simplest and most explicit approach** for this project given:
- The marker class approach (Option A) requires a maintenance burden: updating the marker class every time an ID is added
- `HasVogenConversion()` works directly in entity configuration files the project already has
- It aligns with the "per-type explicit config" philosophy the user chose

However, if Option A (`RegisterAllInVogenEfCoreConverters`) eliminates `StronglyTypedIdConvention` entirely with minimal code, it should be preferred. **Research finding: Option A is cleanest, but requires a marker partial class.** See Code Examples section.

### Pattern 3: Call-site Migration

**What:** All code using `new XxxId(guid)` constructor syntax must change to `XxxId.From(guid)`.

```csharp
// BEFORE
new ProductId(request.Id)
new CategoryId(request.CategoryId)

// AFTER
ProductId.From(request.Id)
CategoryId.From(request.CategoryId)
```

**21 call sites identified** across application commands, queries, and handlers (all in `MicroCommerce.ApiService`). None in domain entities themselves (they use `XxxId.New()` or `XxxId.From()`).

### Pattern 4: EventId in BuildingBlocks.Common

EventId is special: it lives in `BuildingBlocks.Common`, which currently has no Vogen dependency. The migration adds Vogen there.

Current:
```csharp
public record EventId(Guid Value) : StronglyTypedId<Guid>(Value)
{
    public static EventId New() => new(Guid.NewGuid());
    public static EventId From(Guid value) => new(value);
}
```

After:
```csharp
[ValueObject<Guid>(conversions: Conversions.SystemTextJson)]
public partial record struct EventId
{
    public static Validation Validate(Guid value) =>
        value != Guid.Empty
            ? Validation.Ok
            : Validation.Invalid("EventId cannot be empty.");

    public static EventId New() => From(Guid.CreateVersion7());
}
```

Note: EventId does **not** need `Conversions.EfCoreValueConverter` because it is not an EF entity property — it lives only in domain events and is not persisted directly. Only `Conversions.SystemTextJson` is needed if DomainEvent is ever serialized.

### Anti-Patterns to Avoid

- **Using `Conversions.Default` on the attribute:** Default includes TypeConverter, NewtonsoftJson, and more. Use explicit `Conversions.EfCoreValueConverter | Conversions.SystemTextJson` to generate only what's needed.
- **Using `readonly` keyword on the `record struct`:** Vogen generates the struct body; the `readonly` modifier may conflict. The generated code handles immutability.
- **Forgetting `partial` keyword:** The struct will fail to compile — Vogen's generated partial file won't merge.
- **Using `new XxxId(guid)` constructor post-migration:** Vogen's parameterless constructor is `[EditorBrowsable(Never)]` and represents an uninitialized state. The only safe construction is `XxxId.From(guid)` which triggers validation.
- **Calling `New()` instead of `From()` when receiving an ID from external input:** `New()` creates fresh IDs; use `From()` when deserializing existing Guids.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Value object constructor validation | Custom abstract base + validator | `Vogen.Validate()` static method | Vogen hooks validation into `From()` atomically; custom guards can be bypassed via other code paths |
| EF Core value converter for IDs | Reflection-based `IModelFinalizingConvention` | `Vogen.EfCoreValueConverter` nested class | Generated converter is type-safe, handles value comparer correctly for change tracking |
| JSON serialization for IDs | `JsonConverter<T>` subclass | Vogen's generated `SystemTextJson` converter | Handles `ReadAsPropertyName`/`WriteAsPropertyName` for dictionary key scenarios automatically |
| Preventing primitive obsession | Wrapper record with value property | `[ValueObject<Guid>]` | Zero boilerplate, compile-time analyzer enforces correct usage |

**Key insight:** Vogen generates code that handles edge cases (uninitialized struct detection, value comparer for EF tracking, property-name JSON roundtripping) that hand-rolled code typically misses or gets subtly wrong.

## Common Pitfalls

### Pitfall 1: Struct Default Value Bypass

**What goes wrong:** `record struct` has a compiler-generated parameterless constructor. Vogen marks it `[EditorBrowsable(Never)]` but code can still call it. An uninitialized `ProductId` will throw `ValueObjectValidationException` when `.Value` is accessed.
**Why it happens:** `struct` types in C# always have a default zero-value state.
**How to avoid:** Never use `default(ProductId)`, `new ProductId()`, or collection initializers that produce zero-value structs. Always use `ProductId.From(guid)` or `ProductId.New()`.
**Warning signs:** `ValueObjectValidationException: Value object is not initialized` at runtime.

### Pitfall 2: EF Core Primary Key with Struct IDs

**What goes wrong:** EF Core needs to detect sentinel values (null/zero) to determine when to generate a key vs. use the provided key. Struct IDs never have null, so EF Core may not auto-generate keys.
**Why it happens:** EF Core's value generation detection relies on reference-type nullability or specific sentinel detection.
**How to avoid:** Explicitly generate IDs in domain entity constructors using `XxxId.New()` before calling `SaveChanges`. The project already does this (e.g., `ProductId.New()` in `Product.Create()`), so this is not a regression risk.
**Warning signs:** EF Core tries to use a database-generated key (SERIAL) but the ID is already set, causing duplicate key errors or incorrect behavior.

### Pitfall 3: Migration Churn from Converter Type Change

**What goes wrong:** EF Core snapshots record the ValueConverter type name. Switching from `StronglyTypedIdConvention`'s anonymous lambda converter to Vogen's `EfCoreValueConverter` class may change what the snapshot records, triggering unnecessary migration generation.
**Why it happens:** EF Core serializes converter metadata in the model snapshot.
**How to avoid:** After the code migration, run `dotnet ef migrations add VogenIds --context XxxDbContext` for each DbContext and verify the generated migration is **empty** (only an empty `Up()`/`Down()` pair, or no migration needed at all). If the migration is non-empty but contains only converter metadata changes (no table/column alterations), it's safe to generate and apply.
**Warning signs:** Non-empty migration after the code-only change.

### Pitfall 4: `new XxxId(guid)` Constructor Calls Bypassing Validation

**What goes wrong:** Old code uses `new ProductId(request.Id)`. After migration to Vogen, this constructor is discouraged. The compiler won't error on it (it's generated), but validation runs on `From()`, not on the record's primary constructor.
**Why it happens:** Vogen marks the default constructor `[EditorBrowsable(Never)]` but can't prevent record's own constructor from being called.
**How to avoid:** Systematically replace all `new XxxId(value)` with `XxxId.From(value)`. The 21 identified call sites must all be updated.
**Warning signs:** If validation is skipped, an empty Guid passes through.

### Pitfall 5: BuildingBlocks.Common TargetFramework

**What goes wrong:** `BuildingBlocks.Common.csproj` has no `<TargetFramework>` specified — it inherits from `Directory.Build.props` which sets `net10.0`. Vogen 8.0.4 targets `net9.0` and `netstandard2.0`, both of which are compatible with `net10.0` consumers.
**Why it happens:** TFM mismatch between library and consumer.
**How to avoid:** No action needed — `net10.0` consumes `net9.0`/`netstandard2.0` packages without issue. Vogen has no runtime dependencies.
**Warning signs:** Build error `NETSDK1138` or NuGet compatibility warning.

### Pitfall 6: MassTransit Message Contracts

**What goes wrong:** MassTransit saga contracts (in `Contracts.cs`) use primitive `Guid` for `OrderId`, `ProductId`, etc. These do **not** use typed IDs — they're over-the-wire message contracts. Vogen IDs must not be used in MassTransit messages.
**Why it happens:** Cross-process serialization requires primitive types or explicitly registered converters.
**How to avoid:** MassTransit contracts are already using `Guid` — no change needed. The migration only affects in-process domain code.
**Warning signs:** If someone changes a contract to use `ProductId` typed struct, MassTransit STJ serialization may break.

## Open Questions

1. **Does `RegisterAllInVogenEfCoreConverters()` require the marker class to be in the same assembly as `BaseDbContext`?**
   - What we know: The generated extension method name includes the marker class name. The marker class must be `partial` and Vogen generates the body.
   - What's unclear: Whether Vogen scans all assemblies or just the current one for the marker class.
   - Recommendation: Place `VogenEfCoreConverters.cs` in `MicroCommerce.ApiService` alongside `BaseDbContext`. If it fails, fall back to per-property `HasVogenConversion()` in entity configurations.

2. **Will switching from a reflection-based convention to Vogen's typed converter trigger EF Core migration?**
   - What we know: The DB schema (uuid columns) does not change. IDs were already stored as Guid.
   - What's unclear: Whether EF Core's model snapshot records the converter implementation class name.
   - Recommendation: After migration, run `dotnet ef migrations add VogenIds` for all 8 DbContexts and inspect. If migration is empty, no snapshot update is needed. If migration is non-empty but schema-only (no column changes), generate and apply it. This is expected behavior and is documented in Common Pitfalls #3.

3. **Does `BuildingBlocks.Common` need a `<TargetFramework>` entry in its csproj after adding Vogen?**
   - What we know: The csproj currently has no explicit `<TargetFramework>`, inheriting `net10.0` from `Directory.Build.props`. Vogen 8.0.4 targets `net9.0`/`netstandard2.0`.
   - What's unclear: Whether the source generator runs correctly without an explicit TFM in the project.
   - Recommendation: Try adding Vogen without modifying the csproj. If the source generator fails to run, add `<TargetFramework>net10.0</TargetFramework>` explicitly.
