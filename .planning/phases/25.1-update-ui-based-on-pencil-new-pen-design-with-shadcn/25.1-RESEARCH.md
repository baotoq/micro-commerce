## Summary

This phase is a comprehensive visual refresh of the entire MicroCommerce frontend (10 storefront/admin screens + global theme setup) to match the split per-page pencil design files. The work is purely presentational -- no new backend features, API endpoints, or routes are added. Existing React component file names and prop interfaces are preserved where possible; only the JSX/styling gets rewritten.

The current codebase uses Next.js 16.0.3, React 19.2, Tailwind CSS v4.1.17, and shadcn/ui (new-york style, v3.8.5 CLI). There are 18 shadcn/ui components already installed, 23 storefront components, and a well-structured app router layout. The main changes are: (1) swap the color system from default shadcn grayscale to the design's blue-primary palette in oklch, (2) replace Geist Sans with DM Sans font, (3) add custom semantic status variables, (4) add missing shadcn components (tabs, pagination, breadcrumb, checkbox, slider, sidebar), and (5) rewrite each screen's JSX to match the design layout.

**Primary recommendation:** Implement as 11 sequential plans (theme first, then each screen in customer-journey order). Each plan reads its `.pen` design file, installs any needed shadcn components, and rewrites the screen's components to match the design. The theme/globals plan is the foundation -- all subsequent screens depend on it.

### Anti-Patterns to Avoid
- **Don't change hook logic during visual refactor:** Hooks (useCart, useCheckout, useOrders, etc.) remain untouched. Only JSX and className change.
- **Don't hardcode colors as hex in JSX:** Always use CSS variables via Tailwind utilities (`bg-primary`, `text-muted-foreground`). The design's hex values go into globals.css only.
- **Don't pre-install all shadcn components:** Install only what the current plan's screen needs. Avoids unused code and makes it clear which screen introduced which component.
- **Don't break existing route structure:** No file renames, no new routes, no moving pages. This is visual-only.
- **Don't import from design/ at runtime:** .pen files are design-time reference only, never imported in application code.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tab panels | Custom tab state management | `shadcn tabs` (@radix-ui/react-tabs) | Keyboard nav, ARIA roles, focus management |
| Pagination | Custom page number logic | `shadcn pagination` | Proper prev/next/ellipsis, a11y |
| Breadcrumbs | Custom trail rendering | `shadcn breadcrumb` | Semantic nav, separator, ARIA |
| Checkbox groups | Raw `<input type="checkbox">` | `shadcn checkbox` (@radix-ui/react-checkbox) | Indeterminate state, ARIA, consistent styling |
| Range slider | Custom drag handler | `shadcn slider` (@radix-ui/react-slider) | Touch/pointer events, keyboard, ARIA |
| Sidebar navigation | Custom CSS sidebar | `shadcn sidebar` | Responsive collapse, keyboard, sheet on mobile |
| oklch color conversion | Manual math | oklch.com or CSS Color Level 4 tools | Human error in manual conversion; tooling is fast and accurate |
| Font loading | `<link>` tags in `<head>` | next/font/google DM_Sans | Auto-optimization, no CLS, self-hosting, CSS variable output |

**Key insight:** shadcn provides accessible, keyboard-navigable, ARIA-compliant primitives. Hand-rolling these for a visual refresh is wasted effort and introduces accessibility regressions.

## Common Pitfalls

### Pitfall 1: Primary Color Becomes Blue But shadcn Defaults Assume Grayscale
**What goes wrong:** shadcn's default `--primary` is a near-black grayscale. Switching to blue (#2563EB) means hover states like `hover:bg-primary/90` produce a slightly transparent blue instead of a darker shade. Text contrast ratios may shift.
**Why it happens:** The oklch color space handles lightness differently than hex. 90% opacity of a saturated blue on white background looks different than 90% of near-black.
**How to avoid:** After setting --primary to blue oklch, verify all Button variants visually. If hover states look washed out, define an explicit `--primary-hover` or adjust the opacity value in button.tsx cva variants.
**Warning signs:** Buttons that look "faded" on hover rather than "pressed/darkened."

### Pitfall 2: Font Weight Mismatch Between Design and Code
**What goes wrong:** DM Sans loaded with only weights 400/500/600, but design uses `fontWeight: "bold"` (700) somewhere, causing the browser to synthesize bold from 600 -- looks slightly different.
**Why it happens:** Design files may use "bold" generically. DM Sans 600 (semibold) is the heaviest loaded weight.
**How to avoid:** Audit the .pen files for any fontWeight > 600 before implementation. If weight 700 appears, add it to the next/font/google config. From the design-system.pen data reviewed, all weights are 400/500/600 -- safe.
**Warning signs:** Text appearing slightly thicker or thinner than the design mockup.

### Pitfall 3: @theme inline Variables Not Picking Up New CSS Custom Properties
**What goes wrong:** Adding `--success` to `:root` but forgetting to add `--color-success: var(--success)` to the `@theme inline` block means `bg-success` Tailwind class won't work.
**Why it happens:** Tailwind v4 requires both the CSS custom property definition AND the @theme inline mapping for the utility to be generated.
**How to avoid:** For every new CSS variable added to `:root`, add the corresponding `--color-{name}: var(--{name})` to `@theme inline`. Verify with `bg-success` in a test element.
**Warning signs:** Tailwind classes like `bg-success` producing no styles (silently ignored).

### Pitfall 4: Admin Sidebar CSS Variable Scope Bleeding
**What goes wrong:** Overriding `--sidebar` variables globally affects the storefront sidebar/sheet components too.
**Why it happens:** If sidebar variables are set at `:root` level instead of scoped to the admin layout.
**How to avoid:** Scope admin sidebar overrides to the admin layout wrapper element using inline style or a CSS class selector like `.admin-layout { --sidebar: oklch(...); }`.
**Warning signs:** Storefront mobile drawer/sheet using dark navy colors instead of light theme.

### Pitfall 5: E2E Tests Breaking Due to Selector Changes
**What goes wrong:** Playwright E2E tests use selectors like `a[href*="/products/"]`, `getByRole('heading')`, and `getByPlaceholder(/search/i)`. If component structure changes, selectors break.
**Why it happens:** Visual rewrites can change DOM nesting, element types, or remove/rename elements.
**How to avoid:** Keep semantic HTML elements (headings, links, buttons, inputs). Add `data-testid` attributes if needed. After each screen rewrite, run relevant E2E tests if the Aspire stack is available.
**Warning signs:** E2E locators failing to find elements after component rewrite.

### Pitfall 6: Geist Font CSS Variable Still Referenced
**What goes wrong:** `--font-geist-sans` and `--font-geist-mono` still referenced in globals.css `@theme inline` after font swap, causing fallback to system fonts unexpectedly.
**Why it happens:** Incomplete cleanup of the old font configuration.
**How to avoid:** Search for all references to `geist` in the codebase and replace. Key files: `globals.css` (@theme inline `--font-sans`), `layout.tsx` (import and variable).
**Warning signs:** Body text rendering in system default sans-serif instead of DM Sans.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Tailwind v3 `tailwind.config.js` | Tailwind v4 `@theme inline` in CSS | Tailwind v4 (2025) | Theme defined in CSS, not JS config |
| shadcn `hsl()` color format | shadcn `oklch()` color format | shadcn v4 update (2025) | Perceptually uniform colors |
| shadcn `npx shadcn-ui init` | `npx shadcn@latest add` | shadcn CLI v3 (2025) | Unified CLI command |
| next/font `className` only | next/font `variable` + CSS custom property | Next.js 13+ | Better Tailwind integration |
| Geist Sans (default Next.js) | DM Sans (design requirement) | This phase | Match design spec |

**Deprecated/outdated:**
- `tailwind.config.js` / `tailwind.config.ts`: Not used in Tailwind v4 projects. This project correctly uses `@theme inline` in globals.css.
- `hsl()` format in CSS variables: shadcn v4+ moved to `oklch()`. This project already uses oklch (but with default grayscale values).
- `--font-geist-sans` / `--font-geist-mono`: Will be removed in this phase, replaced by `--font-dm-sans`.

## Open Questions

1. **Exact oklch conversion values**
   - What we know: hex values from design-system.pen for all 30+ variables
   - What's unclear: precise oklch values (the ones in the code examples are close approximations from known Tailwind defaults and manual mapping)
   - Recommendation: During Plan 1 implementation, use oklch.com or a Node.js color conversion library to produce exact values. The approximate values above are good enough for planning.

2. **Product Catalog page structure**
   - What we know: The current homepage page.tsx renders both the hero AND the product grid with filters. The design has a separate `storefront-catalog.pen` file.
   - What's unclear: Whether the catalog is a separate `/products` route or remains on the homepage
   - Recommendation: Keep the current structure (homepage = hero + products). The `storefront-catalog.pen` design informs the filters/grid section of the homepage. If a separate `/products` page is needed, it can be a simple re-export of the grid without the hero.

3. **Auth Login page scope**
   - What we know: Current auth uses Keycloak redirect (no custom login page exists in the app). `design/auth-login.pen` exists.
   - What's unclear: Whether to create a custom login page that wraps the Keycloak redirect or just skip this screen
   - Recommendation: Create a simple branded login page at `/auth/login` or `/login` that shows the design's login UI, with the "Sign In" button triggering `signIn("keycloak")`. This is a new lightweight page, not a full custom auth flow.

4. **DM Sans availability in next/font/google**
   - What we know: DM Sans is a Google Font. next/font/google supports it.
   - What's unclear: Whether the exact name import is `DM_Sans` (underscore-separated)
   - Recommendation: HIGH confidence this works. Google Fonts Next.js import convention replaces spaces with underscores: `import { DM_Sans } from "next/font/google"`. Verified pattern from Context7 docs.
