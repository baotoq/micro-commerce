## 4. Runtime API URL Resolution

### GOOD -- Dual-mode resolution pattern
The `getApiBaseUrl()` function in `/Users/baotoq/Work/micro-commerce/src/MicroCommerce.Web/src/lib/config.ts` correctly reads Aspire-injected environment variables (`services__gateway__https__0` or `services__gateway__http__0`) first, then falls back to `API_URL`. ### GOOD -- `/api/config` route exposes server-side URL to client
The route handler at `/Users/baotoq/Work/micro-commerce/src/MicroCommerce.Web/src/app/api/config/route.ts` wraps `getApiBaseUrl()` in a JSON response, allowing client-side code to discover the API URL dynamically without baking it into the JavaScript bundle.

### GOOD -- Singleton promise pattern for client-side resolution
In `/Users/baotoq/Work/micro-commerce/src/MicroCommerce.Web/src/lib/api.ts`, the `getApiBase()` function (lines 4-17) caches the config fetch in `_configPromise`, ensuring only one HTTP request to `/api/config` regardless of how many API calls are made. This is efficient and avoids a waterfall of config requests.

### CRITICAL -- Client-side API calls bypass the Gateway in K8s
In the K8s deployment, the web frontend's server-side `getApiBaseUrl()` resolves to `http://gateway:8080` (from `services__gateway__https__0` env var in the web deployment). However, the `/api/config` route **exposes this internal cluster URL to the browser**. When client-side JavaScript runs `fetch("http://gateway:8080/api/catalog/products")`, this will fail because `gateway:8080` is a Kubernetes internal DNS name that is not resolvable from the user's browser.

Looking at line 41-42 of the web K8s deployment:
```yaml
- name: services__gateway__https__0
  value: "http://gateway:8080"
```

The `getApiBaseUrl()` returns `"http://gateway:8080"`, and `/api/config` sends this to the browser. The browser cannot reach `gateway:8080`.

**Impact:** All client-side API calls (every `getApiBase()` call in `api.ts` that runs in the browser) will fail in the K8s deployment. This breaks the entire storefront for any interactive features.

**Recommendation:** The `/api/config` route should return a browser-reachable URL (e.g., `http://localhost:38800`) rather than the internal service URL. Either:
1. Add a separate env var like `NEXT_PUBLIC_API_URL` set to the external gateway URL and return that from `/api/config`, OR
2. Return a relative URL (empty string) so client fetches go through the Next.js server which can proxy to the gateway, OR
3. Make `getApiBaseUrl()` aware of the distinction between server-side and client-facing URLs.

### WARNING -- SSR-side `fetch` in `api.ts` uses the same `getApiBase()` as client-side
The server-side rendering code path (line 12-13 of `api.ts`) falls back to `NEXT_PUBLIC_API_URL || "http://localhost:5200"`. In K8s, `NEXT_PUBLIC_API_URL` is not set in the web deployment manifest, so SSR calls from the Next.js server that happen to take this path would hit localhost:5200 (which does not exist in the container). However, for server components that `import { getApiBaseUrl } from "@/lib/config"` directly (using the config.ts function), they would correctly get `http://gateway:8080`. The dual-path approach creates confusion about which resolution function to use server-side.

### WARNING -- No cache-busting or TTL on the config promise
Once `_configPromise` is set (line 6 of `api.ts`), it is never reset. If the API URL changes (for example during a rolling deployment or DNS failover), the client will keep using the stale URL until the page is fully refreshed. For a long-lived SPA session, this could cause silent failures.

### INFO -- The `services__gateway__https__0` env var name contains "https" but points to HTTP
In the web K8s deployment (line 41), the env var is named `services__gateway__https__0` but its value is `"http://gateway:8080"`. While this is just a naming convention from Aspire's service discovery and the code reads both `https__0` and `http__0`, it could confuse operators maintaining the K8s manifests.
